01

Basic Input & Output — The Foundation

What input() actually does: It reads one line from the keyboard (or stdin in assessments), waits for Enter, and always returns a string. That's it. Everything else — splitting, converting — is your job.
📥

Single value input

✓ CORRECT
n = int(input())
# input "5" → n is integer 5
✗ WRONG
n = input()
# n is string "5" — math breaks!
# n + 1 → TypeError
If you skip int(), string "5" + 1 crashes. "5" > "10" is True (lexicographic!) — silent wrong answers.
📥

Multiple values on one line

Input: 3 7
✓ CORRECT
a, b = map(int, input().split())
# a = 3, b = 7  (integers)
✗ WRONG
a, b = input().split()
# a = "3", b = "7"  (strings!)
# or:
a = int(input().split()) # crash
# split() returns a list, not int
map(int, ...) applies int() to every element of split(). Missing this = string math = silent WA.
▶ MINI PRACTICE
Input:  10 20 30
Task:   Print their sum
Answer: print(sum(map(int, input().split())))
Output: 60
📤

Output basics — print()

✓ What you want
print(42)        # 42\n
print("yes")     # yes\n
print(a, b)      # 3 7  (space auto-added)
print(a, b, sep=",")  # 3,7
print(a, end="") # no newline after
✗ Common mistakes
print([1,2,3])   # [1, 2, 3]  ← brackets!
print((1,2))     # (1, 2)     ← parens!
print(1,2,3,sep="") # 123  (no spaces)
# vs expected "1 2 3" → WA
print() adds a newline by default, adds a space between multiple args. Both can cause WA if expected format differs.

All Input Patterns — Complete Reference

📋

Array / list on one line

Input: 1 2 3 4 5
✓ CORRECT
arr = list(map(int, input().split()))
# arr = [1, 2, 3, 4, 5]  — integers
✗ WRONG
arr = input().split()
# arr = ['1','2','3','4','5']
# arr[0] + arr[1] = "12"  NOT 3 !
String list: sum(arr) crashes. Sorting "10" < "9" → wrong order. Always convert unless you explicitly need strings.
🔲

Matrix / 2D grid input

Input: 3 (n=3, then 3 rows)
✓ CORRECT
n = int(input())
mat = [list(map(int, input().split()))
       for _ in range(n)]
# mat[0][0] is top-left integer
✗ WRONG
mat = list(map(int, input().split()))
# reads ONLY first row as flat list!
# mat[row][col] → crash (1D, not 2D)

mat = [input().split() for _ in range(n)]
# 2D but strings — math fails!
Flat list instead of 2D: mat[1][2] crashes with IndexError. String elements: comparisons and math silently wrong.
▶ MINI PRACTICE — Print diagonal sum
Input:
3
1 2 3
4 5 6
7 8 9

Code:
n = int(input())
mat = [list(map(int, input().split())) for _ in range(n)]
print(sum(mat[i][i] for i in range(n)))

Output: 15
🔁

Multiple test cases (the most critical pattern)

Most assessment problems use this. The first line is t (number of test cases). Everything else is per-test data.
✓ CORRECT — always wrap in loop
t = int(input())
for _ in range(t):
    n = int(input())
    arr = list(map(int, input().split()))
    print(sum(arr))
✗ WRONG — no loop
t = int(input())  # reads 3 → ignored
n = int(input())  # reads first test's n
arr = list(map(int, input().split()))
# only runs ONCE, misses 2 test cases → WA
Without the loop: only 1 test case runs. If you process t=3 problems but only output 1 answer, judge marks it Wrong Answer immediately.

💬

Strings — with and without spaces

✓ Full line string
s = input()
# "hello world" → s = "hello world"
# Preserves spaces. Use this for sentences.
✗ Splits the sentence!
s = input().split()
# "hello world" → ['hello', 'world']
# s is now a LIST, not a string!
# s[0] = "hello", s[1] = "world"
✓ Multiple words separately
w1, w2 = input().split()
# "hello world" → w1="hello", w2="world"
✗ Wrong split character
parts = input().split(" ")
# "hello  world" (2 spaces!) → ['hello','','world']
# Extra empty string in list → index errors
split() (no args) handles any whitespace including multiple spaces. split(" ") is literal — one space only. Always use split() unless you need CSV-style.
🔀

Mixed data types on one line

Input: 5 hello — an int and a string together
✓ CORRECT
line = input().split()
n = int(line[0])      # "5" → 5
s = line[1]           # "hello" stays string

# OR: unpack first
a, b = input().split()
a = int(a)
✗ WRONG — map(int, ...) on mixed
a, b = map(int, input().split())
# int("hello") → ValueError: crash!
# map(int,...) tries to convert ALL parts
map(int, ...) applies int() to every token. One non-numeric token → crash. Split manually and convert only the int parts.
♾️

Unknown number of inputs / EOF handling

✓ Read all at once (best for EOF)
import sys
data = sys.stdin.read().split()
# Reads EVERYTHING at once into a list
# index pointer pattern:
idx = 0
t = int(data[idx]); idx += 1
for _ in range(t):
    n = int(data[idx]); idx += 1
    arr = list(map(int, data[idx:idx+n]))
    idx += n
✗ Risky with unknown lines
while True:
    line = input()   # raises EOFError at end!
    # Need try/except around this:
try:
    while True:
        line = input()
except EOFError:
    pass
# Works but fragile vs sys.stdin.read()
sys.stdin.read() is safest for EOF problems. input() in a while loop without try/except crashes when input ends.
📑

Multi-line structured input

Input: n on first line, then n separate lines each with one value
✓ CORRECT — read n times
n = int(input())
arr = []
for _ in range(n):
    arr.append(int(input()))
# Each int on its own line → appended
✗ WRONG — one split()
n = int(input())
arr = list(map(int, input().split()))
# Only reads LINE 2, ignores lines 3..n
# arr has 1 element, not n → crash later
One input() reads one line only. If data spans n lines, you must call input() n times (or use sys.stdin.read()).
03

Fast Input — Avoiding TLE on Large Data

When does this matter? When n ≥ 10⁵ and you're reading a lot of lines. Regular input() is slow on large stdin. The judge may give TLE even if your algorithm is correct.
⚡

sys.stdin.readline — the correct way

✓ Fast input setup
import sys
input = sys.stdin.readline

# Now use input() exactly as before:
n = int(input())
arr = list(map(int, input().split()))
✗ The silent trap — missing .strip()
import sys
input = sys.stdin.readline

s = input()
if s == "abc":   # FAILS!
# readline keeps "\n" at end
# s is actually "abc\n"
# "abc\n" == "abc" → False → wrong branch
Always add .strip() when reading strings with fast input. Integer/float conversion via int() or float() handles the newline automatically, but bare string comparison does not.
SAFE FAST INPUT TEMPLATE
import sys
input = sys.stdin.readline

def solve():
    n = int(input())           # .strip() not needed for int()
    s = input().strip()        # ALWAYS strip when using as string
    arr = list(map(int, input().split()))  # .split() strips newline
    return s, arr
✓ Fastest: read all at once
import sys
data = sys.stdin.buffer.read().split()
# Returns bytes, but split() still works
# Fastest possible stdin read
✓ When to use which
# n < 10^4: regular input() fine
# 10^4 ≤ n ≤ 10^6: use sys.stdin.readline
# n > 10^6 or EOF: use sys.stdin.read()

Driver Code Mastery — How Assessments Actually Work

What is driver code? In many platforms (HackerRank, etc.), the platform writes the input/output code for you. It calls your function with specific arguments and expects a specific return value. You only write the function body. Breaking the signature = instant WA.
🔧

Format 1 — Function must RETURN (driver prints)

The driver code already has print(solve(...)). Your job: return the answer. Never print inside.
✓ CORRECT — return only
def solve(arr):
    return sum(arr)

# Driver (hidden, written by platform):
# result = solve(arr)
# print(result)     ← driver prints
✗ WRONG — print inside + return
def solve(arr):
    print(sum(arr))   # prints "15"
    return sum(arr)   # driver then prints "15" again

# Output:
# 15
# 15    ← double output → WA
✗ ALSO WRONG — print but return None
def solve(arr):
    print(sum(arr))  # prints "15"
    # no return statement → returns None

# Output:
# 15
# None     ← driver prints None → WA
✓ Returning list correctly
def solve(arr):
    return sorted(arr)
# Driver handles how to print it
# Don't print [1,2,3] yourself
Every extra print() inside a return-style function adds an extra line to output. Judge sees wrong number of lines → WA immediately.
🖨️

Format 2 — You must PRINT (no return expected)

✓ CORRECT — print inside
def solve():
    n = int(input())
    arr = list(map(int, input().split()))
    print(sum(arr))   # ← you handle output

# You call it yourself:
solve()
✗ WRONG — return instead of print
def solve():
    n = int(input())
    arr = list(map(int, input().split()))
    return sum(arr)   # returned to nobody

solve()   # return value ignored, nothing printed → WA
🏛️

Format 3 — Class-based (HackerRank style)

TYPICAL PLATFORM PATTERN
class Solution:
    def twoSum(self, nums: list, target: int) -> list:
        # Your code only — platform handles I/O
        seen = {}
        for i, v in enumerate(nums):
            if target - v in seen:
                return [seen[target-v], i]
            seen[v] = i
✗ WRONG — changed signature
class Solution:
    def twoSum(self, arr, t):  # renamed params
        # ...

# Platform calls: sol.twoSum(nums, target)
# Python is flexible but type hints help
# Renaming is fine — CHANGING COUNT is not:

    def twoSum(self, nums):  # removed target!
        # platform passes 2 args → TypeError → crash
✗ WRONG — printing instead of returning
class Solution:
    def twoSum(self, nums, target):
        # ...
        print([i, j])   # platform ignores print
        # platform expects return value → gets None → WA
In class-based problems: never change the method signature (parameter count), never print — always return. Platform code calls your method and handles the output.
⚔️

Format 4 — Competitive / write-everything style

You write everything: read input, solve, print output. Most common in Codeforces-style contests.
STANDARD COMPETITIVE TEMPLATE
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    arr = list(map(int, input().split()))
    # ... your logic ...
    print(*sorted(arr))   # you handle output format

t = int(input())
for _ in range(t):
    solve()
Key: variable scope. Any variables inside solve() reset each test case. If you need state across cases, use global or pass as argument.
♻️

Variable reset between test cases — the silent killer

✗ WRONG — variable leaks between cases
count = 0           # outside loop!
t = int(input())
for _ in range(t):
    n = int(input())
    arr = list(map(int, input().split()))
    for x in arr:
        if x > 0: count += 1
    print(count)
# Test 1: count=3 → prints 3 ✓
# Test 2: count=3+2=5 → prints 5 ✗ (should be 2)
✓ CORRECT — reset inside loop
t = int(input())
for _ in range(t):
    count = 0           # reset each case!
    n = int(input())
    arr = list(map(int, input().split()))
    for x in arr:
        if x > 0: count += 1
    print(count)
# Test 1: count=3 → 3 ✓
# Test 2: count=2 → 2 ✓
Variables accumulate silently across iterations. Test case 1 passes, test case 2 onwards fails. Hard to catch without dry-running.
05

Output Formatting — Most Common WA Source

📤

Printing a list — the #1 formatting mistake

✓ Space-separated (most common)
arr = [1, 2, 3]
print(*arr)
# Output: 1 2 3
✗ Prints Python list repr
arr = [1, 2, 3]
print(arr)
# Output: [1, 2, 3]
# Judge expects: 1 2 3 → WA
✓ Each element on own line
for x in arr:
    print(x)
# Output:
# 1
# 2
# 3
✓ Custom separator
print(*arr, sep=",")
# Output: 1,2,3

print(*arr, sep="\n")
# Same as for loop version
print(*arr) unpacks the list. print(arr) prints its string representation with brackets. One wrong character = WA on strict judges.
🔤

String joining — when you need no spaces

✓ No-space join
arr = ['a', 'b', 'c']
print("".join(arr))
# Output: abc

# For integer list:
print("".join(map(str, [1,2,3])))
# Output: 123
✗ join on integers crashes
arr = [1, 2, 3]
print("".join(arr))
# TypeError: sequence item 0: expected str
# Must convert to str first!
↩️

Newline and trailing space control

ALL print() CONTROL OPTIONS
print("hi")              # "hi\n"   — adds newline (default)
print("hi", end="")      # "hi"     — no newline
print("hi", end=" ")     # "hi "    — space instead of newline
print("a", "b")          # "a b"    — space between args (default sep=" ")
print("a", "b", sep="")  # "ab"     — no separator
print("a", "b", sep="-") # "a-b"    — custom separator
print()                  # "\n"     — blank line
If expected output has no trailing newline and you add one, some strict judges fail. Use end="" on the last print to suppress.
🎯

Formatted numbers — float precision

✓ Fixed decimal places
x = 3.14159
print(f"{x:.2f}")   # 3.14
print(round(x, 2))  # 3.14
print("{:.4f}".format(x))  # 3.1416
✗ Precision mismatch
print(1/3)
# 0.3333333333333333
# Expected: 0.33 → WA

print(int(3.7))
# 3  (truncates, doesn't round!)
# Expected: 4 → WA  (use round())

Essential Built-in Functions — Quick Reference

🔨

map() — apply function to iterable

HOW map() WORKS
# map(function, iterable) → lazy iterator
# MUST wrap in list() to get actual list

nums = list(map(int, "1 2 3".split()))  # [1, 2, 3]
nums = list(map(float, "1.1 2.2".split()))  # [1.1, 2.2]
strs = list(map(str, [1, 2, 3]))        # ['1', '2', '3']

# map() is lazy — doesn't execute until consumed:
m = map(int, ["1","2"])   # nothing converted yet
first = next(m)            # NOW converts "1" → 1
rest = list(m)             # converts "2" → [2]
✓ Correct usage
a, b, c = map(int, input().split())
# Perfectly unpacks 3 integers from one line
✗ Forgetting list()
arr = map(int, input().split())
print(arr)  # 
# Not a list! Can't index arr[0]
✂️

split() — anatomy and traps

SPLIT() COMPARISON TABLE
s = "  hello   world  "

s.split()       # ['hello', 'world']    ← strips AND splits on ANY whitespace
s.split(" ")    # ['', '', 'hello', '', '', 'world', '', '']  ← literal space only!
s.split(",")    # ['  hello   world  '] ← no comma found, full string in list

# CSV input — correct way:
"a,b,c".split(",")   # ['a', 'b', 'c']

# Limit splits:
"a b c d".split(None, 2)  # ['a', 'b', 'c d']  — splits max 2 times
🔧

Type conversion quick reference

CONVERSIONS THAT WORK vs CRASH
# SAFE CONVERSIONS:
int("42")         # 42        ✓
int("42\n")       # 42        ✓  (handles trailing newline)
int("  42  ")     # 42        ✓  (handles spaces)
float("3.14")     # 3.14      ✓
str(42)           # "42"      ✓
bool(0)           # False     ✓
bool("")          # False     ✓
bool("0")         # True!     ← "0" is non-empty string!

# CRASHES:
int("3.14")       # ValueError — use int(float("3.14"))
int("42 abc")     # ValueError — split first!
int("10 20")      # ValueError — can't convert multi-token string
int([1,2])        # TypeError  — can't convert list

# Correct: float then int:
int(float("3.7"))  # 3  (truncates to 3, not rounds)
📦

Other essential built-ins for I/O

QUICK REFERENCE
input()          # reads one line, returns string (no newline)
print(*args)     # prints space-separated, adds newline
len("abc")       # 3  — works on str, list, tuple, dict
ord('a')         # 97 — char to ASCII code
chr(97)          # 'a' — ASCII to char
bin(10)          # '0b1010' — to binary string
int('1010',2)    # 10 — binary string to int
hex(255)         # '0xff' — to hex string
int('ff', 16)    # 255 — hex to int

# Useful for arrays:
sum([1,2,3])     # 6
min([3,1,2])     # 1
max([3,1,2])     # 3
sorted([3,1,2])  # [1,2,3] — returns new list
[3,1,2].sort()   # sorts in-place, returns None!
reversed([1,2,3])  # returns iterator, not list
list(reversed([1,2,3]))  # [3,2,1] — need list()
07

Edge Cases & Traps — What Breaks Correct Solutions

🪤

Trailing newline / whitespace in string comparison

✗ Fails silently with fast input
import sys
input = sys.stdin.readline

s = input()
if s == "YES":      # "YES\n" == "YES" → False!
    print("correct")
# Logic correct, but never enters if block → WA
✓ Always strip when comparing strings
s = input().strip()
if s == "YES":      # "YES" == "YES" → True ✓
    print("correct")
With regular input(), newline is stripped automatically. With sys.stdin.readline, it's NOT. This is one of the most common fast-input bugs.
🕳️

Empty input / empty list

✗ Crashes on empty line
arr = list(map(int, input().split()))
print(arr[0])    # IndexError if input was empty line!

print(max(arr))  # ValueError: max() arg is empty sequence
✓ Defensive — check before access
arr = list(map(int, input().split()))
if arr:
    print(arr[0])
    print(max(arr))
else:
    print(0)   # or whatever the problem specifies
1️⃣

Single element edge case

✗ Assumes 2+ elements
arr = list(map(int, input().split()))
# Input: "5"  (single element)
print(arr[1])   # IndexError: list index out of range

a, b = map(int, input().split())
# Input: "5"  (only 1 value)
# ValueError: not enough values to unpack
✓ Check length first
arr = list(map(int, input().split()))
if len(arr) >= 2:
    print(arr[1])
else:
    print(-1)  # handle single element case
🔢

Integer overflow — Python vs other languages

Good news: Python integers have no overflow. 10**100 works fine. However, if a problem says "answer modulo 10^9+7", you must apply it yourself or get WA from wrong value.
MOD PATTERN
MOD = 10**9 + 7
result = (a * b) % MOD       # ✓ apply at each step
result = a * b % MOD         # ✓ same (% has lower precedence)
result = (a + b) % MOD       # ✓
# Don't compute huge number then mod once at end
# (can still be wrong if intermediate values matter)
🔡

Case sensitivity trap

✗ Case mismatch → WA
s = input()  # user types "Yes"
if s == "YES":   # "Yes" != "YES" → False → WA
    print("ok")
✓ Normalize before compare
s = input().strip().upper()
if s == "YES":   # always uppercase → correct

Real Assessment Traps — Simulated Scenarios

🎭

Scenario A — Hidden test cases with different structure

You pass sample test cases but fail hidden ones. Almost always an input-parsing issue.
PROBLEM: "First line is t. Each test case: n, then n numbers."
Sample input visible:          Hidden input:
2                              5
3                              1
1 2 3                          2
2                              3
4 5                            4
                               5
                               1 2 3 4 5
✗ Wrong — assumes array on same line as n
t = int(input())
for _ in range(t):
    line = input().split()
    n = int(line[0])
    arr = list(map(int, line[1:]))  # tries to get arr from same line!
    # Hidden input has n alone on its line → arr is empty → WA
✓ Correct — n on its own line
t = int(input())
for _ in range(t):
    n = int(input())        # n is alone
    arr = list(map(int, input().split()))  # array is next line
    print(sum(arr))
🎭

Scenario B — You solve it, but output has wrong separator

PROBLEM: Sort and print. Expected: "1 2 3". Got: "1\n2\n3"
arr = [3, 1, 2]

# What you wrote (each on new line):
for x in sorted(arr):
    print(x)
# Output: 1\n2\n3  → WRONG if expected "1 2 3"

# Fix:
print(*sorted(arr))
# Output: 1 2 3 ✓

# Or if expected "1,2,3":
print(",".join(map(str, sorted(arr))))
# Output: 1,2,3 ✓
Always read output format in the problem statement. "space-separated" ≠ "one per line" ≠ "comma-separated".
🎭

Scenario C — Debug print left in code

✗ Debug print ruins output
def solve(arr):
    print("DEBUG arr =", arr)  # ← forgot to remove!
    return sum(arr)

# Output:
# DEBUG arr = [1, 2, 3]
# 6
# Judge sees 2 lines, expects 1 → WA
✓ Use stderr for debug (doesn't affect stdout)
import sys
print("DEBUG arr =", arr, file=sys.stderr)
# Only shows in your terminal, judge ignores stderr
🎭

Scenario D — Consuming input out of order

Input: t on line 1, then each test case is: string on line 1, n on line 2, array on line 3
WRONG CODE:                        CORRECT CODE:
t = int(input())                   t = int(input())
for _ in range(t):                 for _ in range(t):
    n = int(input())  ← reads s!       s = input().strip()
    s = input().strip()← reads n!      n = int(input())
    arr = list(map(int,                arr = list(map(int,
          input().split()))                   input().split()))
# All values shifted → garbage            # Correct order ✓
Read inputs in the exact order they appear. One swap cascades — every subsequent read gets wrong data.
🎭

Scenario E — YES/NO vs Yes/No vs yes/no

READ THE EXPECTED OUTPUT FORMAT!
# Problem says output "YES" or "NO"
print("Yes")   # ← WA (wrong case)
print("yes")   # ← WA
print("YES")   # ✓

# Problem says output "True" or "False"
print(True)    # prints "True" ✓
print("true")  # WA (lowercase)

# If in doubt, copy the exact string from problem statement

⚡ Top 15 WA Causes — Know These Cold

TRAP 01

Missing test case loop

Only 1 of t test cases runs. All other cases ignored → guaranteed WA.

TRAP 02

print(arr) instead of print(*arr)

Outputs [1, 2, 3] with brackets. Judge expects 1 2 3.

TRAP 03

No .strip() with fast input

String comparison fails: "abc\n" == "abc" is False.

TRAP 04

split(" ") instead of split()

Two spaces → empty string token in list → index errors or wrong values.

TRAP 05

print inside + return

Driver also prints → answer appears twice → WA.

TRAP 06

Not resetting variables per loop

Values accumulate across test cases → correct for case 1, wrong for rest.

TRAP 07

Reading input in wrong order

One swap cascades → every subsequent read gets mismatched data.

TRAP 08

Forgetting list() on map()

map() is lazy. Printing it gives <map object> or crashes on index.

TRAP 09

String list instead of int list

Missing int(): "10" < "9" is True → sorting, comparison, math all wrong.

TRAP 10

Debug print left in code

Extra output line. Strict judge fails immediately on unexpected tokens.

TRAP 11

map(int,...) on mixed types

int("hello") crashes. Split manually and convert only numeric parts.

TRAP 12

Wrong YES/NO case

Problem says "YES", you print "Yes". Case mismatch = WA.

TRAP 13

Changed class method signature

Platform passes 2 args, you defined 1 → TypeError → runtime crash.

TRAP 14

int() on float string

int("3.14") crashes. Use int(float("3.14")) instead.

TRAP 15

Not applying modulo

Problem says "answer mod 10^9+7". You return raw value → WA on large inputs.
